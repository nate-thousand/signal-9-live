# Signal 9 Visual Language

Signal 9 is the operating system of The Resistance. The player is not using an app. The player is operating an illegal analog communications network inside a world controlled by Project Digital Harmony.

This document defines the visual operating philosophy for every Signal 9 screen, animation, component, visual effect, and future design prompt. It is not a UI redesign. It is the standard that future UI work must obey.

## Core Philosophy

Music is civilization.

Memory is resistance.

Analog systems preserve individuality.

Digital systems optimize conformity.

Signal 9 exists to recover, protect, and rebroadcast human cultural memory. Every recovered transmission, Memory Node, vinyl record, broadcast, Echo, and field recording strengthens humanity against systems that flatten identity into predictable behavior.

The interface must communicate that the player is working with fragile, powerful, repairable equipment. It should feel built by engineers, archivists, radio operators, field agents, musicians, and dissidents. It should not feel like consumer software.

## Visual Philosophy

Signal 9 is equipment, not software. Screens are instruments. Panels are modules. Controls are diagnostic surfaces. Motion is signal behavior. Visual noise is evidence of transmission, not decoration.

Every interface decision should answer one question:

Does this make the player feel like they are operating an illegal analog communications network under hostile surveillance?

If not, remove it or reframe it.

## Fictional Operating Context

The world is controlled by Project Digital Harmony, a conformity system that optimizes culture into predictable, obedient signal. Signal 9 resists by preserving analog memory: music, broadcasts, physical recordings, oral histories, corrupted archives, and human voices that cannot be fully normalized.

The UI is the player’s field terminal. It is made from:

- Radio engineering
- Broadcast equipment
- Analog communications
- Military avionics
- CRT terminals
- Industrial instrumentation
- Telemetry
- Signal intelligence
- Electronic warfare
- Oscilloscopes
- Packet analyzers
- Communications diagnostics

The UI must never resemble:

- Consumer software
- SaaS dashboards
- Mobile apps
- Glassmorphism
- Modern web applications
- Lifestyle productivity tools
- Social feeds
- Decorative cyberpunk skins

## Art Direction Summary

Signal 9 is black-box resistance hardware: terminal glass, field radios, rack-mounted broadcast decks, signal analyzers, tape machines, avionics displays, damaged CRTs, and improvised surveillance countermeasures.

Visuals should feel:

- Engineered
- Functional
- Modular
- Repairable
- Analog
- Broadcast driven
- Military grade
- Industrial
- Technical

Every panel should feel like equipment instead of software. Every animation should represent signal behavior instead of decorative motion.

## Typography

Typography should feel like terminal firmware, military labels, signal readouts, equipment markings, and broadcast diagnostics.

Use type to distinguish operational roles:

- Display type: system identity, screen titles, boot states, emergency labels.
- Mono type: command input, telemetry, logs, IDs, frequencies, coordinates, packet-like data.
- Compact labels: control labels, panel headers, status chips, equipment annotations.
- Dense readouts: diagnostics, spectrum values, signal health, mission state.

Avoid typography that feels editorial, lifestyle-oriented, rounded-consumer, or marketing-led. Text should feel stamped, encoded, printed, terminal-rendered, or instrument-labeled.

## Grid System

Signal 9 should use an instrumentation grid, not a content marketing layout.

The grid should imply:

- Rack units
- Broadcast console bays
- Avionics panels
- Oscilloscope divisions
- Terminal rows
- Diagnostic columns
- Modular repairable cards

Recommended structure:

- Primary frame: persistent system boundary and status rail.
- Work bay: central operational area for transmission, mission, or memory work.
- Side modules: deck, diagnostics, inventory, lore, radio, Echo, or signal controls.
- Footer telemetry: continuous machine state, not decorative navigation.

Grids may be dense, but they must remain legible. Whitespace should feel like unused panel space, not luxury spacing.

## Panel Layout

Panels are equipment modules. They should have clear operational purpose.

Every panel needs:

- A machine-readable title or code.
- A visible state: active, standby, degraded, locked, recovered, transmitting, archived.
- A role in the system: monitor, control, archive, input, output, diagnostic, tuner.
- A relationship to signal: source, route, amplifier, decoder, memory, warning, log.

Avoid generic cards. If a panel cannot be described as a piece of equipment, it should not exist.

## Motion Language

Motion represents signal behavior:

- Tuning
- Lock-on
- Drift
- Interference
- Scan
- Pulse
- Packet arrival
- Tape or reel movement
- Transmission acquisition
- Carrier loss
- Buffering as analog recovery, not spinner loading

Motion should be purposeful, mechanical, and tied to system state. Decorative hover animations should be minimized unless they communicate a change in signal, focus, or hardware readiness.

## Audio Reactive Language

Audio reactivity is not a visualizer skin. It is the machine exposing that music is active cultural memory.

Audio reactive behavior should feel like:

- Signal gain
- Carrier strength
- Oscilloscope response
- Magnetic tape saturation
- Broadcast compression
- Interference bursts
- Low-frequency power draw
- Memory recovery pulses

Bass can drive weight, density, pulse, or power. Mids can drive motion, carrier stability, or modulation. Highs can drive noise, flicker, scanlines, or packet sparkle. Peaks can cause glitch, warning flash, distortion, or brief overload.

## ASCII Language

ASCII is Signal 9’s living display material. It is not nostalgia for its own sake. It is low-bandwidth cultural persistence: a form that survives hostile networks, dead screens, lossy transfers, and corrupted archives.

ASCII should communicate:

- Recovered image data
- Improvised rendering
- Low-bandwidth resistance technology
- Human-readable machine output
- Memory reconstructed from damaged signal

ASCII can be used for:

- Startup identity
- Transmission previews
- Video-to-ASCII stage output
- Mission glyphs
- Signal diagnostics
- Corrupted portraits
- Memory Node previews
- Echo spectral signatures

Avoid using ASCII as random decoration. It should be tied to signal, memory, identity, or diagnostics.

## Image Processing Rules

Images are recovered transmissions, not glossy media.

Image treatment should prioritize:

- Dither
- Scanline
- Threshold
- Contrast recovery
- Bloom as signal overload
- Chroma loss
- VHS or RF interference
- Compression artifacts
- Ghost frames
- Partial corruption
- Metadata overlays
- Signal provenance

Image processing should imply that the system is reconstructing memory from hostile or damaged channels. Avoid clean hero images unless the fiction says the signal is unusually pristine.

## Terminal Language

Terminal copy is operational. It should sound like a field system, not a chatbot UI.

Use terms like:

- `TRANSMISSION`
- `CARRIER`
- `LOCK`
- `ECHO`
- `NODE`
- `MEMORY`
- `RELAY`
- `BAND`
- `FREQUENCY`
- `ARCHIVE`
- `DECODE`
- `INTERFERENCE`
- `SIGNAL`
- `GHOST`
- `PROJECT DIGITAL HARMONY`

Prompts should feel like commands, diagnostics, recovered logs, or radio procedure. Avoid conversational helper language unless it belongs to GHOST’s voice.

## Animation Principles

Animation should obey these rules:

- It must represent system state, signal behavior, mechanical response, or memory recovery.
- It should have a source: audio, user input, transmission state, network state, mission state, or AI state.
- It should degrade gracefully under reduced motion.
- It should be legible under visual noise.
- It should not compete with the ASCII stage.
- It should feel like hardware reacting, not software delighting.

## Interaction Principles

Interaction should feel like operating equipment.

Controls should imply:

- Switches
- Tuners
- Faders
- Deck transport
- Patch routing
- Signal gain
- Frequency selection
- Record / transmit / monitor
- Diagnostic toggles

Feedback should be immediate and stateful: lock indicators, meter movement, carrier warnings, log entries, signal pulses, or command echoes.

## Component Principles

Components are modular hardware units. Each component should have an operational role and an accountable state.

Preferred component families:

- Broadcast deck
- Frequency selector
- Signal meter
- Telemetry strip
- Mission panel
- Memory Node card
- Echo signature
- Radio tuner
- Inventory module
- Packet log
- Diagnostic overlay
- Transmission queue
- Recovery progress indicator

Avoid components that feel like generic web cards, modal dialogs, marketing accordions, or mobile bottom sheets unless they are visually reframed as equipment.

## Color Philosophy

Color is signal state, not decoration.

Use color to express:

- Carrier strength
- System health
- Interference
- Warning
- Memory recovery
- Faction or source identity
- Transmission type
- AI influence
- Archive status

The dominant field should remain dark and restrained. Bright colors are earned by signal events, warnings, recovered memory, or active transmission. Purple/violet can remain Signal 9’s core spectral identity, but it should feel like phosphor, RF glow, or cultural signal, not brand polish.

## Information Density

Signal 9 can be dense because it is instrumentation. Density should be structured, not cluttered.

High density is appropriate for:

- Telemetry
- Logs
- Diagnostics
- Signal analysis
- Mission state
- Inventory metadata
- Memory archive records
- Radio frequencies

Low density is appropriate for:

- Startup moments
- Major transmission events
- Memory recovery reveals
- Mission completion
- Echo encounters

Density should breathe according to signal importance.

## Environmental Storytelling

Every UI surface can tell the story of resistance infrastructure.

Use:

- Wear through labels, not random grime
- Serial numbers
- Relay IDs
- Station callsigns
- Archive stamps
- Frequency marks
- Hand-repaired module names
- Warning codes
- Signal provenance
- Corruption history
- Transmission timestamps

Avoid decorative lore dumps. The system should imply a world through operational details.

## Signal Processing Effects

Signal effects should be meaningful and tied to state:

- Static: interference, surveillance pressure, weak carrier.
- Glitch: hostile digital intrusion or overloaded decode.
- Scanlines: CRT/monitor surface, not a decorative overlay.
- Bloom: signal gain or overload.
- Chromatic offset: channel drift or damaged source.
- Jitter: unstable lock.
- Dropout: carrier loss.
- Feedback: recursive transmission loop.
- Thresholding: decoder sensitivity.
- Dither: low-bandwidth reconstruction.

Every effect should be explainable as a signal process.

## Broadcast UI Patterns

Broadcast UI should center on source, carrier, route, output, and archive.

Common patterns:

- `NOW TRANSMITTING`
- Deck transport
- Carrier lock meter
- Source selector
- Frequency band
- Transmission queue
- Live log
- Signal health strip
- Audio-reactive meter
- Archive capture marker
- Relay routing display

The player should understand that broadcasts are acts of resistance.

## Mission Interface Patterns

Mission UI should feel like operational tasking, not a quest list.

Use:

- Field orders
- Relay objectives
- Signal intercepts
- Recovery targets
- Threat state
- Route status
- Transmission constraints
- Extraction conditions

Mission state should feel live, unstable, and affected by transmissions.

## Transmission Interface

Transmission interfaces should focus on acquiring, decoding, stabilizing, and rebroadcasting signal.

Key states:

- Searching
- Acquiring
- Locked
- Decoding
- Degraded
- Jammed
- Rebroadcasting
- Archived
- Lost

Transmission UI should make the player feel that sound and memory are moving through a dangerous network.

## Inventory Interface

Inventory is not a backpack. It is a field archive and equipment manifest.

Inventory items should be treated as:

- Physical media
- Signal tools
- Keys and codes
- Recovered devices
- Memory artifacts
- Broadcast parts
- Evidence

Inventory UI should include provenance, condition, signal relevance, and operational use.

## Memory Node Interface

Memory Nodes are cultural memory fragments. They are not collectibles.

A Memory Node should communicate:

- Source
- Time
- Medium
- Integrity
- Cultural value
- Recovery status
- Related Echoes
- Related transmissions

The interface should make recovery feel like preservation, not acquisition.

## Echo Interface

Echoes are traces of people, songs, places, and moments that survived digital suppression.

Echo UI should feel spectral but technical:

- Waveform fragments
- Voiceprint-like signatures
- ASCII silhouettes
- Corrupted captions
- Signal fingerprints
- Memory resonance

Avoid magical ghost UI. Echoes are emotional because they are recovered through machines.

## Radio Interface

Radio UI should feel like tuning across a forbidden spectrum.

Use:

- Frequency bands
- Callsigns
- Station IDs
- Carrier strength
- Squelch
- Scan
- Fine tune
- Interference maps
- Relay path
- Broadcast status

Radio should be tactile, technical, and central to the fiction.

## Developer Guidelines

When adding UI, ask:

- What equipment is this?
- What signal does it monitor or control?
- What state does it expose?
- How does it support music, memory, resistance, or analog survival?
- Does it avoid consumer software patterns?
- Does it preserve Signal 9 theme ownership?
- Does it avoid modifying engines unless explicitly scoped?

Prefer app-owned semantic tokens and Design System bridges over copied platform tokens. Keep Signal 9’s visual identity in the app.
